## Approach vs. the plan

Before writing the final script I drove the actual editor (manually started `npm run serve`,
scripted the same page with Playwright, dumped DOM state) to verify the plan's selectors against
the real Motion Canvas 3.17 UI, per the task's instruction to treat the plan's code as a starting
point, not gospel. Two real bugs in the plan's approach surfaced and were fixed:

1. **Resolution-field manipulation was unreliable as written.** The plan sets `el.value` via a
   plain property assignment + dispatches synthetic `input`/`change` events. In testing, this
   silently failed for the width field while appearing to work for the height field (order- and
   timing-dependent ŌĆö a reactivity quirk of the Motion Canvas UI's signal-based number input, not
   something to paper over). Fix: use Playwright's `locator.fill()` (real focus/select-all/type/blur)
   on `input[type=number]` index 2 (width) and index 3 (height), with a short 200ms pause between
   the two fills. Verified repeatedly stable in both directions (1920ŌåÆ1080 and 1080ŌåÆ1920, and vice
   versa) across multiple runs. The runner also reads back both values after filling and throws if
   they don't match `1080`/`1920` ŌĆö no silent pass-through on a DOM regression.
2. **Completion detection via "does `output/project.mp4` exist yet" is wrong and was actually
   observed to fail.** The FFmpeg exporter opens/creates the output file the moment rendering
   starts and keeps writing to it for the render's full duration ŌĆö "file exists" was true only
   ~2.5s into what turned out to be a ~15s render. Worse: when I first closed the browser early in
   exploration (before understanding this), the encoder process was left mid-stream with nothing
   feeding it more frames, and it hung indefinitely holding an OS-level lock on the file (had to
   `taskkill` a stuck `ffmpeg.exe` manually to unblock further testing). Fix: the render button
   (`#render`) gains a `data-rendering` attribute while `RendererState.Working` (confirmed by
   reading `@motion-canvas/core`'s `Renderer.render()`/`RendererState` source) and loses it again
   once the renderer returns to `Initial` (success, error, or abort). The runner waits for that
   attribute to appear (render actually started, 15s timeout) and then waits for it to disappear
   (render actually finished, 5 min timeout) before ever checking for the file. This was verified
   stable across two independent clean end-to-end runs (~14.7s and comparable on the second run).
3. **Windows process-tree cleanup was missing and caused a real orphan.** `spawn('npm', [...],
   {shell: true})` on Windows returns the PID of the wrapping `cmd.exe`; a plain `child.kill()`
   only kills that shell, not the actual `vite`/node process or the `ffmpeg` child it spawns. I hit
   exactly this during exploration ŌĆö port 9000 stayed bound and a background `node.exe` kept
   running after a script exited. Fix: `killServerTree()` uses `taskkill /pid <pid> /T /F` on
   `win32` (falls back to a process-group `SIGKILL` on POSIX). Verified after both real
   `npm run render` runs: `netstat` shows port 9000 free and no `ffmpeg.exe` process remains.

Selectors that matched the plan exactly and needed no change: the FFmpeg exporter `<select>`
(found by matching option text `/ffmpeg/i`, then `selectOption({label: 'Video (FFmpeg)'})`), and
`page.getByRole`-equivalent targeting of the render button (used `page.locator('#render')` since
the button has a stable `id="render"` ŌĆö more robust than role/name matching).

## Concerns
- Full render takes ~15s wall-clock (vite boot + browser render), well within reasonable CI/local
  bounds, but the 5-minute completion timeout is a guess for longer specs (up to 3 scenes per the
  schema) ŌĆö should be revisited if future scene-specs run noticeably longer.
- Task 3's report flagged port 9000 sometimes being occupied by other processes on this shared dev
  machine. I confirmed port 9000 was free before both integration runs and free again afterward;
  `render-runner.mjs` still hardcodes `PORT = 9000` with no fallback if something else is squatting
  on it when Task 5/6 run this ŌĆö `waitForServer` will just time out after ~30s in that case with a
  clear `dev server did not start` error, not a silent hang.
- The `.meta` files committed alongside the runner encode the 1080x1920/ffmpeg-exporter settings as
  the project's persisted defaults. This is convenient but means the runner's own explicit
  set-resolution/set-exporter steps are technically redundant after the first run ŌĆö left them in
  intentionally since idempotent explicit configuration is safer than relying on persisted local
  editor state (e.g. a fresh clone / CI checkout won't have arrived at that state through the UI).
